# 乱数生成の仕組み

ルーレットの停止位置は、**ボタン押下時刻（`Date.now()`）をシードにした mulberry32 PRNG** で決定している。このドキュメントは「なぜ `Math.random()` ではなく seeded PRNG を使っているか」「当選判定はどう行われるか」を、再現可能な粒度でまとめたもの。

関連コード: [src/script.js:14-24](src/script.js#L14-L24)（PRNG）, [src/script.js:116-134](src/script.js#L116-L134)（spin 本体）。

## なぜ seeded PRNG か

`Math.random()` でも統計的には十分に一様だが、実装が以下の事情でブラウザ依存・不透明である：

- ブラウザごとにアルゴリズム（xorshift128+、PCG など）や初期シードの取り方が異なる。
- シード値に介入できないので、**同一入力に対する再現**ができない（デバッグ・検証が難しい）。
- SpiderMonkey / V8 はシードをプロセス起動時に決めるため、連続スピンで「近い時間に呼ばれた `Math.random()` が近い値を返す」可能性が理論上ゼロではない（実測レベルでは無視可能だが、根拠を説明しづらい）。

そこで、**シードをこちらで制御できる決定的 PRNG（mulberry32）** に差し替え、シードには毎回違う値となる `Date.now()`（ミリ秒精度の UNIX 時刻）を使う構成にしている。結果として以下のメリットが得られる：

1. **アルゴリズム固定**。どのブラウザでも同じシードから同じ乱数列が出る。
2. **完全再現可能**。`Date.now()` の戻り値さえ分かれば、当選項目を後から検証できる。
3. **シードが毎回ユニーク**。連打しても `Date.now()` は最低 1ms ずれるので、シードは重複しない。

## アルゴリズム: mulberry32

32bit 状態・周期 2³² の軽量 PRNG。擬似乱数としては amateur-grade だが、ルーレットの停止角度を決める用途（0〜359 の均等な整数が得られれば十分）には過剰なほどの品質がある。

```js
function mulberry32(seed) {
    return function () {
        let t = seed += 0x6D2B79F5;
        t = Math.imul(t ^ t >>> 15, t | 1);
        t ^= t + Math.imul(t ^ t >>> 7, t | 61);
        return ((t ^ t >>> 14) >>> 0) / 4294967296;
    };
}
```

- 戻り値は `[0, 1)` の浮動小数（32bit 精度）。
- 同じ seed を与えれば何度でも同じ値を返す。
- ルーレットでは **1 スピンにつき 1 回しか呼ばない**（`rand()` を 1 回だけ実行）。周期・相関は実質問題にならない。

## 停止角度の決定

```js
const rand = mulberry32(Date.now());
const extra = Math.floor(rand() * 360);   // 0..359
currentRotation += (360 * 10) + extra;    // 10 周 + 追加角度
```

- `currentRotation` は**累積値**。前のスピンからの見た目の連続性を保ったまま、毎回 10 周分以上回し切る。
- `extra` の分布は `rand()` が一様なので、0〜359° に均等分布する。
- スピン時間は CSS の `transition: transform 5s` に依存（[src/styles.css:20](src/styles.css#L20)）。JS 側は 5.1 秒後に結果を確定（[src/script.js:131](src/script.js#L131)）。

## 当選項目の計算

```js
const actual = (360 - (currentRotation % 360)) % 360;
const idx = Math.floor(actual / (360 / items.length));
selectedItem = items[idx];
```

- `currentRotation % 360` は「ホイールが時計回りに何度回ったか」。
- 針は上（12 時方向）固定なので、**針から見た角度** は `360 - 回転角` に相当（逆方向の相対角）。
- その角度をスライス幅 `360 / items.length` で割った商が、12 時から時計回り 0, 1, 2... と番号付けしたスライスのインデックス。
- `items.length` が 360 の約数でなくても問題ない（`Math.floor` で切り捨て、`extra < 360` が保証されているのでインデックス範囲外にはならない）。

## 均等性の担保

`extra` は 0〜359 の 360 通りを（事実上）一様にとるので、項目数 N での各スライスの選択確率は:

- N が 360 の約数（例: 8, 10, 12, 18）: 完全に 1/N。
- N が約数でない（例: 7, 11, 13, 17）: `floor(360/N)` 度のスライスと `ceil(360/N)` 度のスライスが混在し、最大で `1/360` ≒ 0.28% の偏りが出る。実用上は無視してよい。

## 既知の制限・注意点

- **シードの予測可能性**: `Date.now()` はユーザの手元で観測できる。悪意あるユーザが押下時刻を厳密に制御すれば、特定の項目が出るまで試行できる。本プロジェクトは「出目を当てるゲーム」ではなく「公平な抽選」が目的なので、この予測可能性は問題にしない。秘匿が要件になったら `crypto.getRandomValues()` に切り替える。
- **ミリ秒同時クリック**: `isSpinning` フラグでスピン中の多重クリックは弾くので（[src/script.js:117](src/script.js#L117)）、同一ミリ秒で 2 回シードが決まるケースは発生しない。
- **時計の手動変更**: ユーザが PC の時計を戻せば同じシードを再現できるが、これも「結果を事前に知って当てる」攻撃シナリオに対する懸念に過ぎず、抽選の公平性には影響しない。

## 変更する場合の注意

- `mulberry32` を差し替える際は、**戻り値が `[0, 1)` の一様分布**であることを保証すれば `spin()` 側は触らなくていい。
- シード源を変える場合（例: `performance.now()` + `crypto.getRandomValues()` ミックス）、**ミリ秒単位の一意性と単調性** だけは保っておくと連打時の挙動が予測しやすい。
- `(360 * 10)` の `10` はアニメーション演出上の「最低 10 周」で、確率分布には影響しない。スピン時間（CSS の 5s）を変えるならここも合わせて調整する。
