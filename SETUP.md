# セットアップ・デプロイ手順書

初回セットアップから公開までの手順と、運用開始後の更新手順・トラブルシューティング。

## 構成のおさらい

```
[ブラウザ] ──GET──▶ https://<user>.github.io/<repo>/    (フロント: GitHub Pages)
     │
     ├──GET──▶  GAS ウェブアプリ (/exec)                 (項目取得)
     └──POST─▶  GAS ウェブアプリ (/exec)                 (ログ保存, text/plain)
                        │
                        ▼
                  Google スプレッドシート
                        ├── ルーレット シート（入力）
                        └── ログ シート（出力）
```

セットアップは 3 ステップ：**① スプレッドシート → ② GAS バックエンド → ③ GitHub Pages フロント**。

---

## 前提

- Google アカウント（スプレッドシート + GAS 用）
- GitHub アカウント（フロント配信用）
- このリポジトリのローカルクローン（= このフォルダ）
- `git` コマンド
- 任意のランダム文字列生成手段（例: macOS なら `openssl rand -hex 16`）

---

## Part 1. スプレッドシートの準備

1. Google ドライブで新しいスプレッドシートを作成（または既存のものを利用）。
2. 左下のシートタブをダブルクリックして、シート名を `ルーレット` に変更。
3. A 列に以下のように入力：

   | | A |
   | :--- | :--- |
   | 1 | マリミラルーレット（← ページタイトル。自由に変更可） |
   | 2 | 当たりA |
   | 3 | 当たりB |
   | 4 | 当たりC |
   | … | … |

4. `ログ` シートは初回の保存時に自動生成される。手動で作る必要はない。

---

## Part 2. GAS バックエンドのデプロイ

### 2-1. Apps Script プロジェクトを開く

1. 手順 1 で作成したスプレッドシートで、**拡張機能 → Apps Script** を開く。
2. 左サイドバーの `コード.gs`（または `Code.gs`）をクリックし、中身を全削除。
3. ローカルの [Code.gs](Code.gs) をエディタで開き、内容をコピーして GAS エディタに貼り付け。
4. <kbd>Ctrl/Cmd + S</kbd> で保存。

> ファイル名は GAS 上も `コード.gs` のままで OK（内容で判定される）。

### 2-2. API_TOKEN をスクリプトプロパティに登録

1. GAS 左サイドバー下部の **⚙️ プロジェクトの設定** をクリック。
2. 「スクリプト プロパティ」の「スクリプト プロパティを編集」をクリック。
3. 「プロパティを追加」：
   - **プロパティ**: `API_TOKEN`
   - **値**: ランダム文字列（例: `openssl rand -hex 16` の出力。32 桁の 16 進数）
4. 「スクリプト プロパティを保存」をクリック。

> このトークンは Part 3 でフロント側にも同じ値を設定するので、**メモしておく**こと。

### 2-3. ウェブアプリとしてデプロイ

1. 右上の **デプロイ → 新しいデプロイ** をクリック。
2. 歯車アイコン → **ウェブアプリ** を選択。
3. 以下を設定：
   - **説明**: `initial`（任意）
   - **実行ユーザー**: `自分`
   - **アクセスできるユーザー**: `全員`
4. 「デプロイ」ボタンをクリック。
5. 初回は権限確認ダイアログが出るので：
   - 「アクセスを承認」→ 自分の Google アカウントを選択
   - 「このアプリは Google で確認されていません」が出たら「詳細」→「（安全ではないページ）に移動」
   - 権限リスト（スプレッドシートへのアクセス等）を確認して「許可」
6. デプロイ完了画面で **ウェブアプリの URL** が表示される。これをコピー。

> URL は `https://script.google.com/macros/s/XXXXX/exec` の形式。**Part 3 で使う**。

### 2-4. バックエンドの動作確認

コピーした URL をブラウザで直接開く。以下のような JSON が返れば OK：

```json
{"title":"マリミラルーレット","items":["当たりA","当たりB","当たりC"]}
```

エラーが出る場合は [トラブルシューティング](#トラブルシューティング) を参照。

---

## Part 3. フロントエンド（GitHub Pages）のデプロイ

### 3-1. フロントの設定を書き換え

1. [src/script.js](src/script.js) をエディタで開く。
2. 冒頭の設定ブロックを編集：

   ```js
   const API_URL = 'https://script.google.com/macros/s/XXXXX/exec';  // ← Part 2-3 でコピーした URL
   const API_TOKEN = 'abcdef1234567890...';                           // ← Part 2-2 で設定したトークン
   ```

3. 保存。

### 3-2. ローカルで動作確認（任意だが推奨）

GitHub にプッシュする前にローカルで検証：

```sh
cd "<このフォルダ>"
python3 -m http.server 8000
```

ブラウザで `http://localhost:8000` を開く。

**期待される挙動**：
- ページタイトルが「マリミラルーレット」（スプレッドシートの A1）になる
- SVG ルーレットが項目数分の色で描画される
- 「ルーレットを回す」ボタンが活性化する
- ボタン押下 → 回転 → モーダル表示 → 名前入力 → 保存 → スプレッドシートの `ログ` シートに 1 行追加される

> ローカルでは `localhost` 扱いで Geolocation が動く。GPS は `"未取得"` のまま保存される場合もあるが API 動作確認としては十分。

### 3-3. GitHub リポジトリに push

まだローカルが git 管理されていない場合：

```sh
cd "<このフォルダ>"
git init
git add .
git commit -m "Initial commit"
```

GitHub でリポジトリを作成（空リポジトリ・README なしで OK）してから：

```sh
git remote add origin https://github.com/<username>/<repo>.git
git branch -M main
git push -u origin main
```

> **注意**: `src/script.js` に `API_TOKEN` がベタ書きされているため、パブリックリポジトリにすると誰でも読める。bot よけ程度の効果として運用するか、プライベートリポジトリ（GitHub Pro が必要）を使うか判断する。

### 3-4. GitHub Pages を有効化

1. GitHub でリポジトリを開く。
2. **Settings → Pages**（左サイドバー）。
3. 「Build and deployment」で：
   - **Source**: `Deploy from a branch`
   - **Branch**: `main` / `/ (root)`
4. 「Save」をクリック。
5. 数十秒〜数分待つ。ページ上部に「Your site is live at `https://<username>.github.io/<repo>/`」が表示されたら公開完了。

### 3-5. 本番動作確認

公開 URL をスマホのブラウザで開く：

- ページタイトル / ルーレットが表示される
- スピンして保存 → 位置情報の許可ダイアログが出たら **許可** をタップ
- スプレッドシートの `ログ` シートに、日付・時間・項目・名前・緯度・経度の 1 行が追加される

## 運用: 更新手順

### フロント（HTML/CSS/JS）を変更したとき

```sh
git add .
git commit -m "..."
git push
```

push 後、数十秒で GitHub Pages に反映される。ブラウザキャッシュが効いていれば強制リロード（<kbd>Shift + F5</kbd> / <kbd>Cmd + Shift + R</kbd>）。

### バックエンド（Code.gs）を変更したとき

1. GAS エディタで内容を貼り付け直して <kbd>Ctrl/Cmd + S</kbd>。
2. 右上 **デプロイ → デプロイを管理** をクリック。
3. 既存デプロイ右の **鉛筆アイコン（編集）** をクリック。
4. 「バージョン」のドロップダウンで **新しいバージョン** を選択。
5. 「デプロイ」をクリック。

> **必ず「デプロイを管理」から既存デプロイを更新する**。「新しいデプロイ」を作ると URL が変わり、フロントの `API_URL` も更新が必要になる。

### API_TOKEN をローテーションしたいとき

1. 新しいランダム文字列を生成。
2. GAS: プロジェクトの設定 → スクリプト プロパティ → `API_TOKEN` の値を新しい文字列に更新して保存。
3. フロント: [src/script.js](src/script.js) の `API_TOKEN` を同じ文字列に書き換え。
4. `git commit && git push`。

GAS 側は再デプロイ不要（`PropertiesService` は即時反映）。フロントは GitHub Pages が反映するまで数十秒待つ。

## トラブルシューティング

| 症状 | 原因と対処 |
| :--- | :--- |
| ブラウザで「読み込み失敗」アラート | `API_URL` が間違っている／GAS デプロイ未完了／アクセス権限が「自分のみ」になっている。GAS URL を直接ブラウザで開いて JSON が返るか確認 |
| GAS URL を直接開くと Google のログイン画面が出る | デプロイ設定の「アクセスできるユーザー」が `全員` になっていない。デプロイを管理から修正 |
| 保存しても `ログ` シートに行が増えない | ブラウザ DevTools の Console に `保存失敗:` が出ていないか確認。`unauthorized` が返っていれば `API_TOKEN` 不一致 |
| 「このアプリケーションは Google Apps Script のユーザーによって作成されたものです」の警告ページが出る | GAS の URL を**直接**ブラウザで開いた場合に出る仕様。フロント（GitHub Pages 側）を開けば出ない |
| GitHub Pages が 404 | Pages 設定後 1〜2 分待つ。それでもダメなら main ブランチ直下に `index.html` があるか確認 |
| 位置情報が `"許可なし"` で記録される | ブラウザのサイト設定から位置情報を「許可」に変更。iOS Safari は設定 → Safari → 位置情報 も確認 |
| 位置情報が `"未取得"` のまま | 5.1 秒のスピン時間内に GPS 取得が間に合わなかった。屋内・地下では時間がかかる。通常再試行で改善 |
| 再デプロイ後に保存が失敗するようになった | 誤って「新しいデプロイ」してしまい URL が変わった可能性。「デプロイを管理」から新しい URL を確認し、[src/script.js](src/script.js) の `API_URL` を更新 |
| CORS エラー (`No 'Access-Control-Allow-Origin'`) が Console に出る | `Content-Type: application/json` で POST していないか確認。必ず `text/plain;charset=utf-8` を使う（`application/json` は GAS が応答できない preflight を発生させる） |
